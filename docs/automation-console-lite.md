# AI 自动化控制台 Lite

## 定位

AI 自动化控制台 Lite 是一个轻量级自动化平台框架，用来管理零散的业务自动化工具。

它不是单一脚本，也不是一开始就追求大厂级复杂平台。当前目标是：

```text
一个稳定主系统
多个独立工具
统一配置
统一调度
统一日志
失败可追溯
工具可替换
```

## 当前使用模式

```text
飞书 = 团队提交入口
本机 Docker = 自动化处理平台
后台 = 管理员控制台
Git = 工具和平台变更记录
```

团队成员不直接进入后台。团队只在飞书多维表格提交任务，后台监听飞书任务表并处理。

## Lite 版职责

平台核心负责：

- 模块管理
- 工具目录扫描
- 配置管理
- 工作流触发
- 任务日志
- 运行历史
- 错误状态
- 基础降级

工具负责：

- 具体业务逻辑
- 外部 API 调用
- 文件处理
- AI 能力调用
- 工作流步骤实现

## 不在 Lite 版里做的事

当前阶段暂不引入：

- Kubernetes
- 多租户
- 插件市场
- 复杂 RBAC 权限
- 分布式任务队列
- 微服务拆分
- 多语言 Worker 平台

这些能力等真实任务量、用户数和维护成本上来后再升级。

## 参考方向

Lite 版借鉴的是大厂平台的简化思想：

- Backstage：统一门户、工具目录、插件化
- Airflow / Conductor：工作流、步骤状态、失败重试
- Argo Workflows：未来每个重任务可拆成独立执行单元

当前项目不是复制这些系统，而是保留它们的核心结构：

```text
platform core + tool registry + workflows + logs + isolated tools
```

## 演进路线

### Phase 1: 本机 MVP

- FastAPI + Next.js
- SQLite
- Docker Compose
- 飞书任务表作为团队入口
- 管理后台只给管理员使用

### Phase 2: 可控内测版

- 后台登录
- API 鉴权
- CORS 限制
- 上传大小和行数限制
- 日志脱敏
- 密钥加密存储

### Phase 3: 团队服务器版

- HTTPS
- 反向代理
- 防火墙
- PostgreSQL
- Redis + Worker
- 备份和恢复

### Phase 4: 平台化版本

- 工具运行时隔离
- 工作流定义外置
- 任务队列
- 多环境配置
- 更细权限和审计

## 判断标准

新增功能前先问：

1. 这是平台核心能力，还是某个工具能力？
2. 能不能放进 `backend/tools/<tool_id>/`？
3. 是否通过 capability 调用，而不是直接写死模块？
4. 失败后是否只影响当前工具？
5. 删除这个工具会不会影响其他工具？

如果答案不清楚，先不要写进主系统。
