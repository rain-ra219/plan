# 面试讲解稿

这份文档用于把项目讲清楚。重点不是背稿，而是帮助你形成稳定表达。

## 30 秒版本

我做了一个 AI 自动化后台平台 MVP，用来管理一些零散的自动化工具。它不是单一脚本，而是一个控制台：支持模块启停、配置管理、工作流运行、日志追溯和失败降级。

目前实现了一个完整闭环：上传 CSV 后，系统会清洗线索、按客户归并、同步到飞书多维表格，并记录上传历史和每一步任务日志。飞书失败时，流程会标记为部分成功，数据仍然保留在本地；如果启用了消息通知模块，还可以通过 `message.send` webhook 发出异常通知。

## 2 分钟版本

这个项目的背景是：实际工作里会有很多零散自动化工具，比如飞书同步、线索清洗、客户归并、图片生成、文案生成等。它们经常需要迭代和替换，所以我没有把业务流程写死成某个固定脚本，而是做了一个平台化的后台。

平台本体负责通用能力：

- 用户操作入口
- 模块管理
- 配置管理
- 工作流运行
- 任务日志
- 状态看板
- 失败降级

业务能力通过模块接入，并抽象成 capability。比如流程里不是直接写“调用飞书”，而是调用 `table.write`，当前由飞书模块实现，未来可以替换成数据库、CRM 或 Notion。

第一版 MVP 我做了一个真实业务闭环：

```text
上传 CSV -> 线索清洗 -> 客户归并 -> 同步飞书线索明细表 -> 同步飞书客户表 -> 查看上传历史和任务日志
```

这个闭环里有几个关键点：

- 线索按“来源平台 + 询盘时间 + 客户名称 + 商品标题 + 原始咨询内容”去重。
- 客户 ID 优先使用联系方式，没有联系方式时用客户名称 + 地区。
- 一个客户可以有多条线索，客户表只做汇总，不丢失具体问题。
- 飞书同步使用本地记录和飞书 record_id 的映射，重复上传时会更新记录，不会重复新增。
- 飞书失败时，工作流状态是 `partial_success`，并保留本地数据和错误日志。
- 消息通知是独立的 `message.send` 模块，通知失败不会影响主流程。

## 5 分钟演示结构

1. 仪表盘：展示平台可以监控任务运行状态。
2. 功能管理：展示模块化、启停、健康状态。
3. 配置中心：展示飞书配置和密钥管理。
4. CSV 上传：上传样例文件并运行工作流。
5. 上传历史：展示处理行数、线索数、客户数、飞书新增/更新数。
6. 任务日志：展示每个能力调用都有日志。
7. 数据中心：展示线索明细和客户归并结果。
8. 降级演示：停用飞书模块，再跑一次，展示 `partial_success`。
9. 通知演示：启用消息通知模块，展示 `message.send` 日志。

## 面试官可能问的问题

### 为什么不直接写一个飞书同步脚本？

因为脚本只能解决单点问题，但实际工作里工具会不断变化。今天是飞书，明天可能是 CRM、Airtable 或 Notion。如果直接写死调用飞书，后续替换成本很高。

所以我把流程抽象成能力调用，比如 `table.write`。飞书只是当前能力提供方，未来替换模块时，工作流本体可以尽量少改。

### 这个项目最核心的设计是什么？

核心是“平台本体”和“业务模块”分离。

平台本体负责调度、配置、日志、状态和降级。业务模块只负责实现具体能力，比如清洗线索、归并客户、写入飞书。

### 怎么处理飞书失败？

飞书失败不会让整个流程失败。系统会把本地线索和客户先保存下来，然后把飞书写入步骤记录为失败或跳过，整体工作流标记为 `partial_success`。

这样业务人员至少不会丢数据，开发人员也能通过任务日志看到具体失败原因。

### 消息通知模块有什么意义？

它证明这个平台不是只有飞书一个模块。飞书同步属于 `table.write` 能力，消息通知属于 `message.send` 能力。

当工作流出现 `partial_success` 或 `failed` 时，系统会尝试通过 webhook 发通知。通知模块没有启用、没配置或者发送失败，都只会记录到任务日志，不会影响主工作流结果。

### 怎么避免重复同步到飞书？

系统有一张 `external_record_mappings` 表，记录本地记录 ID 和飞书 record_id 的关系。

第一次同步时创建飞书记录，并保存映射。再次上传同一条线索或客户时，系统会找到对应 record_id，调用飞书更新接口，而不是新增一条重复记录。

### 这个项目现在还不完善的地方是什么？

当前还是 MVP，下一步可以继续加强：

- 把更多模块改成真正独立的 HTTP/MCP 插件。
- 增加用户权限和操作审计。
- 增加后台任务队列，比如 Celery + Redis。
- SQLite 切换到 PostgreSQL。
- 密钥加密存储。
- 增加自动化测试和 CI。

### 你在这个项目里体现了什么能力？

我觉得主要体现了四类能力：

- 业务抽象：把零散脚本抽象成模块和 capability。
- 后端工程：FastAPI、SQLite、工作流状态、日志追踪、外部 API 集成。
- 前端工程：Next.js 控制台、模块管理、配置中心、数据展示。
- 部署交付：Docker Compose、一键启动脚本、演示文档和故障降级。
